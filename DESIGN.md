# 설계 기록 — 불편한 시간만 알려주세요

토스 Product Designer Challenge 2026 · 회의 시간 조율 솔루션 (참석자 화면)

## 1. 문제 정의

**표면 문제:** 6명이 겹치는 빈 시간을 못 찾는다.

**근본 문제 (과제 원문에 정박):** 과제 문장의 동사는 "동료들의 상황을 **이해하고**", 목표는 "모두가 **괜찮은** 시간"이다.
회의를 못 잡는 진짜 이유는 각자의 제약이 **캘린더에 없고, 주관적이며, 말하기 곤란**해서 — 주최자가 상황을 **이해할 수단이 없기** 때문이다.
특히 "점심 직후는 웬만하면 피하고 싶다" 같은 **약한 선호**는 말로 꺼내면 단호한 요구처럼 과장되므로(까다로워 보일까 봐) 아예 삼켜진다. → 주최자의 안개는 여기서 생긴다.

**JTBD (주최자·지연):** 막막함 없이, '괜찮은' 시간을 **확신을 갖고** 정한다.
**JTBD (참석자):** 이 조율에 **최소한으로 끌려들고** 싶다. 내 사정은 반영되되, 노출·책임·눈치는 지고 싶지 않다.

## 2. 경험 문법 (전 화면을 지배하는 원칙)

1. **막막함 → 선명함.** 답만 주지 말고 상황이 이해되게 — *상황(나)을 거쳐야 답(가)을 믿는다.*
2. **이진이 아니라 정도(gradient).** "가능/불가능"은 약한 선호를 표현 못 한다. 입력(슬라이더 세기)부터 출력(괜찮음의 정도)까지 하나의 문법.
3. **컴퓨터가 아는 것 vs 나만 아는 것.** 캘린더에 있는 건 시스템이 자동 반영. 참석자는 *캘린더에 없는, 나만 아는 사정*만 표현한다.
4. **말하지 않고 흘린다.** 비언어(슬라이더) · 사적(아무에게도 안 보임) · 정당화 불요(항목은 미리 깔려 있음) · 부인 가능(요구가 아니라 기울기).
5. **인간 판단은 유지, 기계 작업은 위임.** 필수/선택은 주최자가 정하고(정치적 판단), 대조·계산은 시스템이 한다.

## 3. 페르소나 (장면 기반)

| 인물 | 역할 | 핵심 텍스처 |
|---|---|---|
| **지연** | 주최(PO) | 캘린더가 아니라 '얼굴들'로 생각함. 라운드를 돌아도 아는 게 안 늘어 막막함. 감으로 후보를 던짐 |
| **수아** | 필수 | 점심 직후 싫지만 말하면 유난스러울까 봐 삼킴 → "이른 시간 안 될까요?"로 우회, 이유는 증발 |
| **민수** | 필수 | 화요일 외근 잦음(캘린더에 안 잡힘). 매 라운드 말로 반복 설명당함 |
| **최분석** | 선택 | 답해야 하나 눈치, 나 때문에 못 잡히는 부담. 지연도 부르기 미안함 |

## 4. 솔루션 구조

- **입력 = 미리 깔린 흔한 걸림돌 팔레트** (시간대 3종 + 요일). 수아가 짓는 게 아니라 **알아보고 당기기만**.
- **세기 4단계:** `편해요(기본·무의견) · 살짝 피하고 싶어요 · 웬만하면 피하고 싶어요 · 안 돼요(MAX=거절·하드)`
  - MAX만 단어·색(빨강)이 다름 — 선호와 불가의 경계를 말투로 드러냄.
  - 소프트 구간은 모두가 밀어도 답이 사라지지 않게 하는 안전장치.
- **요일도 같은 축.** "화요일 발표라 월요일은 웬만하면 피하고 싶다" 같은 소프트 요일이 실존 → 요일 탭 → 그 요일의 슬라이더 (한 번에 하나만 편집 → 높이 불변).
- 필수/선택 = 주최자가 정한 **가중치**. 선택 참석자에겐 안심 배너("꼭 안 와도 괜찮아요").

## 5. UX 라이팅 결정 (토스 5대 밸류·8원칙 + corca 명확성 원칙 적용)

| 요소 | 확정 카피 | 근거 |
|---|---|---|
| 헤더 | 불편한 시간만 알려주세요 | 조건 먼저, "살짝"으로 최소노동 암시 |
| 캘린더 칩 | ✓ 캘린더에 있는 일정은 이미 빼뒀어요 | 자동 반영의 신뢰 신호. 결과 명시 |
| 서브 | 괜찮으면 그냥 넘겨도 돼요 | 안 해도 됨을 숨기지 않음 |
| 기본 라벨 | **편해요** | "좋아요"는 안 한 판단을 과잉주장, "괜찮아요"는 미지근, "가능해요"는 능력 축 혼입 → 마음 축 유지 |
| MAX 라벨 | 이 시간은 **안 돼요** | 선호(피하고 싶어요)와 단어를 다르게 → 경계 표시 |
| CTA | 입력 있음 **다 정했어요** / 없음 **다 편해요** | "다 됐어요"는 "다 돼요(전부 가능)"와 충돌. "알려줄게요/보낼게요"는 대인·전송 프레임이라 지운 청중을 되살림 → 자기완결 |
| 프라이버시 | 🔒 고른 내용은 아무에게도 안 보여요 · 시간 정하는 데만 써요 | 수아의 청중 공포 해소 |
| 완료 | 다 정했어요! 시간이 정해지면 가장 먼저 알려드릴게요 | Hidden Emotion + Predictable Hint |
| 선택자 배너 | 꼭 안 와도 괜찮아요. 부담 없이 편하게 알려주면 돼요 | "선택 참석"의 눈치를 입력 전에 해소. 배제 아닌 안심 톤 |
| 이탈 모달 | 지금 나가시겠어요? / 지금까지 고른 내용은 저장되지 않아요 | 결과 명시. primary=계속하기(안전) |

**삭제한 것 (근거와 함께):**
- ~~MAX 경고 박스("정말 안 되는 시간에만…")~~ → "안 돼요" 라벨+빨강으로 이미 3중 전달(중복). 제약을 표현한 순간에 훈계 = 심리적 안전 파괴. 정직성 가드는 필요 시 **제출 시점** 1회로.
- ~~행마다 "편해요/안 돼요" 양끝 라벨~~ → 섹션 우상단 범례 1회로.
- ~~When2meet식 격자·채팅 입력·4단계 무의식 캡처~~ → 각각 이진 함정·번역 함정·프로파일링 불신.

## 6. 화면 구조 (토스 문법: 한 화면 한 스텝, 스크롤 없음)

```
Step 1 시간대  : 이탈< · 컨텍스트 · 타이틀 · 캘린더칩 → 슬라이더 3 (아침→점심→오후)
                → [이 중에 없는 사정이 있어요] [다음]
Step 2 요일    : 이탈< · 타이틀 → 요일 스트립 + 단일 에디터(탭한 요일만)
                → 프라이버시 · [이전][다 정했어요/다 편해요]
완료           : 체크 · "다 정했어요!" · 알림 예고
이탈 모달      : 바텀시트 [나가기][계속하기] (1:1)
```

**내비 의미론:** 좌상단 ＜ = 플로우 이탈(확인 모달 경유) / 스텝 이동 = CTA 좌측 보조 버튼.

**쇼케이스 쉘:** 회색 바닥 + 그림자 프레임(=카드). 우상단 토글 2개(참석 유형/디바이스), 양옆 드래그 리사이즈(px 뱃지), 720px 경계에서 토글 자동 동기화. 반응형은 **컨테이너 쿼리**(iframe 없음). 실기기(≤640px)에선 쉘 자동 숨김.

**폴리시:** make-interfaces-feel-better 원칙(동심원 radius, scale .96 on press, 40px 히트영역, transition 속성 명시, 눈금은 채움색에 동화 등).
